2. **Cross-process message inject** (`00b74073` + docs `f69a3b0f`) —
   `crush sessions inject <id> -m/-f [--interrupt] [--json]` lets an
   operator inject a message into a session running in a DIFFERENT
   `crush run` process (the `/crush` sub-agent use case). Design:
   empty `pending_injects` DB table = zero steady-state cost; non-interrupt
   delivery piggybacks on the PrepareStep that already runs every
   provider turn (no new poll loop); interrupt delivery uses a
   lightweight 3s ticker scoped to the active turn's lifetime only.
   `SessionAgentCall.ExistingMessageID` prevents duplicate message rows
   across the CLI-creates/agent-splices boundary. `messages.Notify()`
   pushes live UI updates when a web UI happens to be attached to the
   same process. Built in two waves of `om` sub-agents (foundation,
   then CLI+interrupt-ticker in parallel); orchestrator personally
   verified the full chain end-to-end (traced `ExistingMessageID`
   through both the initial-Run and requeue-after-cancel paths) rather
   than trusting agent self-reports.

3. **deploy.go cross-OS first-install + PATH setup, unit-tested**
   (`73853a60`) — `deploy.go` previously only replaced an
   already-installed crush; now falls back to a standard per-user
   install location (`%LOCALAPPDATA%\Programs\crush\crush.exe` on
   Windows via registry PATH append through PowerShell's
   `[Environment]::SetEnvironmentVariable` — deliberately not `setx`,
   which silently truncates PATH > 1024 chars; `~/.local/bin/crush` on
   Unix, printing an export line if not already on PATH — never edits
   shell rc files). The key design move: extracted all decision logic
   (install path, PATH-membership checks, executable-vs-shim detection,
   cwd-excluding PATH lookup) into a new `internal/deploy` package of
   pure functions with no side effects, covered by
   `internal/deploy/deploy_test.go` (9 tests) — this makes `go test
   ./...` exercise deploy logic on ALL THREE OSes in the existing CI
   build matrix (`ubuntu-latest`, `macos-latest`, `windows-latest`),
   not just whatever OS a human happens to run `deploy.go` on.
   `deploy.go` itself stays a thin `//go:build ignore` wrapper. CI was
   personally verified green on all 3 matrix jobs, not just the
   aggregate.

**Currently open, investigation-only, NOT yet implemented**: the
operator asked (via `/o46h`) to research a new feature — letting the
main agent AND the human operator inject a message into ANY sub-agent
(or ALL sub-agents at once) running under a session, without
interrupting them, from the web UI. Investigation found:

- Sub-agents are already full `session` rows with
  `ParentSessionID` set; sub-session ID is
  `CreateAgentToolSessionID(messageID, toolCallID)` = `"msgID$$toolCallID"`.
  `ListSubSessions(parentSessionID)` already exists and returns all
  children.
- The existing `inject_message` WS command
  (`SendMessagePayload{SessionID, Content}` → `handleInjectMessage` →
  `coordinator.InjectMessage`) is **generic over SessionID already** —
  it will work against a sub-agent's composite session ID with ZERO
  backend changes, because it was never restricted to top-level
  sessions.
- So "inject into one specific sub-agent" needs ONLY a frontend change:
  a mini input inside `SubAgentBlock.tsx` that computes the same
  `${messageID}$$${toolCallID}` key already used for message lookup
  and sends `inject_message` with that as `sessionID`.
- "Inject into ALL sub-agents at once" was scoped as two options:
  (A) frontend loops `$subAgentSessions` map and fires N individual
  `inject_message` calls (simpler, more WS chatter, client-side
  busy-filtering is racy), or (B) one new WS command
  `inject_all_subagents{parentSessionID, content}` where the backend
  does `ListSubSessions` + filters to busy sessions + calls
  `InjectMessage` per child (cleaner, ~30 lines of Go). Orchestrator
  recommended (B) but did not implement either — this was explicitly
  a research-only turn (`/o46h ... исследуй задачу`), the user has not
  yet said "делай" or picked A vs B.
- A stretch idea (tool-based inject where the MAIN agent's own model
  calls a new `inject_subagent` tool programmatically) was named but
  explicitly deferred as "second iteration" — not scoped for this
  round.

## Decisions
- **Sub-agent inject: reuse existing `inject_message` WS command for
  the single-target case** — it is already SessionID-generic, so no
  backend change is needed for that half of the feature. Only the
  broadcast-to-all-subagents half needs new code, and there the
  recommendation is a new backend WS command (option B) over N
  frontend-side individual sends (option A), for atomicity and
  server-side busy-filtering — not yet confirmed by the user.
- **CHANGELOG.fork.md and docs/checkpoints/* are append-only history**
  — reaffirmed multiple times this session (YOLO removal, this
  checkpoint) that these are never edited retroactively even when they
  describe now-removed or now-stale features.
- **Every sub-agent-delegated implementation is personally re-verified
  by the orchestrating session** — not trusting agent self-reports.
  This caught one real cross-package interface-compliance bug (the
  YOLO removal's `mockSessionService` gap) that both delegated agents
  missed because neither touched that file.
- **Pure-logic extraction pattern for testability across OSes**:
  when script-like `//go:build ignore` files need OS-conditional
  logic verified in CI, extract the decision logic (no side effects)
  into a proper `internal/` package covered by `go test`, and let the
  existing CI OS matrix (`build.yml`: ubuntu/macos/windows) provide
  real per-OS coverage instead of hand-rolling emulation. Established
  as a reusable pattern for `deploy.go`, likely applicable to
  `build.go` too if it ever needs similar coverage.
