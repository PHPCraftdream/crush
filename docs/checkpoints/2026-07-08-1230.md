The bulk of this session was a deep, iterative bug hunt triggered by a companion Claude Code session working in `D:\dev\rust\shamir-db` (a separate Rust repo) reporting that `crush run` sessions kept "dying immediately at launch." Root-caused through many false starts to a real chain of Windows-specific bugs, all fixed and shipped to the locally-built/deployed `crush.exe` (via `go run deploy.go`, which rebuilds and replaces the npm-installed binary on PATH):

1. **Peak-hours mid-turn stop** (feature, not bug): `checkPeakHours` was only checked once at turn start; added a re-check in `OnStepFinish` so an in-flight turn stops the moment the provider's `peak_hours` window opens (commits around `agent.go`/`coordinator.go`, predates this checkpoint's visible log window — see `git log` further back than the 5 shown).
2. **Windows session-lock false "crashed"**: `LockFileEx` is a *mandatory* Windows lock — any other process reading the lock file's PID hits a sharing violation for the holder's entire lifetime, so `pid=0` is normal for a live session, not proof of death. Fixed `sessions why`/`sessions list` to fall back to heartbeat freshness (`session.LockStaleDuration`) when PID is unreadable, but still trust a *confirmed*-dead PID unconditionally.
3. **Peak-hours error swallowed on mid-turn abort**: calling `cancelFn()` to break fantasy's agent loop made `agent.Stream()` return generic `context.Canceled` instead of the specific `*PeakHoursError`, and forcing the specific error back in then hit ANOTHER bug where the generic downstream error-handler overwrote the good finish message. Both fixed (`peakHoursAbortErr` stash + explicit `*PeakHoursError` branch in `agent.go`).
4. **`RESUME AT` guidance lost in fang's ERROR box**: fang truncates multi-line error text; the guidance is now printed directly to stderr before the ERROR box, in `app.go`.
5. **Real instant-death root cause**: a detached orchestrator launch (`crush run < p > out 2> err &`, wrapper shell exits instantly) → wrapper's console closes → Windows sends `CTRL_CLOSE_EVENT` to every attached process, including crush, and for THAT event the process is unconditionally terminated after the handler chain returns (a Windows guarantee, not something a ctrl-handler can prevent). Fixed via `cmd.maybeDetachConsole()`: when all three std streams are redirected, `FreeConsole()` at the very start of `run`'s `RunE`.
6. **Side effect of #5**: with crush itself console-less, EVERY child process it spawns (bash-tool commands via mvdan.cc/sh, MCP stdio servers, ripgrep, cliprovider CLI launches, and — the actual biggest offender — `git rev-parse` calls that `internal/config/load.go` runs **unconditionally on every single config load**) got a freshly *visible* console window each, flashing on screen. Traced this down via isolated reproduction (a standalone `npx` spawn test, confirmed clean both with and without a console-attached parent) which *disproved* the "npx grandchildren" hypothesis and pointed at the real, more mundane culprit: unconditional `git rev-parse` at startup. Added `internal/platform.HideConsoleWindow(cmd *exec.Cmd)` (sets `SysProcAttr.HideWindow` on Windows, no-op elsewhere) and applied it at every relevant spawn site: `internal/config/load.go` (git rev-parse x2), `internal/config/docker_mcp.go`, `internal/agent/tools/rg.go`, `internal/agent/tools/mcp/init.go` (MCP stdio spawn + its stdioCheck probe), `internal/agent/cliprovider/{provider,mcpserver}.go`, `internal/shell/exec_windows.go` (both the custom bash-tool exec handler and the shebang-script fallback), `internal/app/run_on_finish.go`. **User confirmed the final clean test showed zero window flashing.**

Throughout, I repeatedly rebuilt and redeployed the local binary via `go run deploy.go` (which force-kills any running `crush.exe` via `taskkill /F` — checked `crush sessions locks` before each redeploy to avoid killing a real in-flight session, and did once collaterally kill an already-stale/orphaned lock, which was fine).
